# Звіт до лабораторної роботи №15

## Тема
Синхронізація потоків у Java.

## Мета роботи
Було набутo практичних навичок роботи зі спільним змінюваним станом у багатопотокових програмах Java. Було розглянуто проблему стану перегонів, визначено критичні секції та використано `synchronized` для коректної роботи кількох потоків зі спільним ресурсом.

## Індивідуальне завдання
Варіант 3: бронювання номерів. Необхідно створити класи `HotelRoom`, `BookingRequest`, `BookingTask` і реалізувати ситуацію, коли кілька потоків одночасно намагаються забронювати обмежену кількість номерів.

## Предметна область
Предметною областю є система бронювання номерів у готелі. У системі є клієнти, номери, заявки на бронювання та створені бронювання. Кілька працівників або користувачів можуть одночасно надсилати заявки на бронювання, тому виникає потреба у синхронізації доступу до спільного пулу номерів.

## Спільний ресурс
Спільним ресурсом є пул доступних номерів:

- список доступних номерів;
- список успішно створених бронювань.

Ці структури змінюються кількома потоками, тому вони належать до спільного змінюваного стану.

## Небезпечний варіант без синхронізації
Клас `UnsafeHotelRoomPool` виконує бронювання без блокування. Операція бронювання складається з кількох кроків:

1. знайти доступний номер потрібного типу;
2. вилучити номер зі списку доступних;
3. змінити стан номера;
4. створити об’єкт бронювання.

Якщо два потоки одночасно знайдуть один і той самий номер до його вилучення, можливий некоректний результат. Це і є стан перегонів.

## Синхронізований варіант
Клас `SynchronizedHotelRoomPool` використовує об’єкт `lock` як монітор:

```java
synchronized (lock) {
    Room room = findAvailableRoom(request.getRoomType());
    if (room == null) {
        return false;
    }
    availableRooms.remove(room);
    room.book();
    reservations.add(new Reservation(...));
    return true;
}
```

У критичній секції одночасно може працювати лише один потік. Завдяки цьому пошук, вилучення номера і створення бронювання виконуються як одна атомарна операція.

## Критична секція і вибір монітора
Критичною секцією є не вся програма, а тільки операція зміни спільного пулу номерів. Для синхронізації використано окремий об’єкт `lock`, а не зовнішню колекцію. Це робить межі блокування зрозумілими і не дозволяє зовнішньому коду випадково синхронізуватися на тому самому об’єкті.

## Демонстраційний сценарій
У класі `Main` запускаються три сценарії:

1. `UnsafeHotelRoomPool` — кілька потоків працюють без синхронізації.
2. `SynchronizedHotelRoomPool` — кілька потоків працюють із синхронізацією.
3. Послідовне виконання — той самий обсяг заявок обробляється в одному потоці.

Для потоків використовується `Thread`, запуск виконується через `start()`, очікування завершення — через `join()`.

## Параметри експерименту
- Кількість потоків: 4.
- Кількість заявок на потік: 100 000.
- Початкова кількість доступних номерів: 1 000.
- Очікувана максимальна кількість успішних бронювань: 1 000.

## Результати виконання
У несинхронізованому варіанті результат може бути нестабільним, оскільки порядок виконання потоків не гарантується. У синхронізованому варіанті кількість успішних бронювань не перевищує кількість доступних номерів, а після завершення всіх потоків стан пулу залишається узгодженим.

## Порівняння послідовного і паралельного виконання
Час виконання вимірюється за допомогою `System.nanoTime()`. Паралельний варіант не завжди швидший, тому що синхронізація має накладні витрати: захоплення монітора, очікування блокування та перемикання потоків. У цій лабораторній роботі важливішою є не максимальна швидкість, а коректність спільного доступу.

## Тестування
У тестах перевірено:

- успішне бронювання одного номера;
- відхилення заявки, якщо номерів більше немає;
- коректний фінальний результат після паралельного синхронізованого сценарію.

## Висновок
Було реалізовано систему бронювання номерів зі спільним ресурсом. Було показано, що операції зі спільним змінюваним станом потребують синхронізації. Несинхронізований варіант може працювати нестабільно, а використання `synchronized` забезпечує коректне виконання критичної секції та узгоджений фінальний стан після завершення всіх потоків.
